# Принятые решения Verona CS2

Краткий журнал нужен, чтобы будущая сессия не отменяла осознанные решения как будто это случайность.

## D001 — собственная реализация вместо форка WeaponPaints

**Решение:** использовать публичный API CounterStrikeSharp и написать минимальный модуль заново.

**Почему:** WeaponPaints включает ненужные меню, MySQL и сторонние плагины, а также лицензируется GPL-3.0. Собственная реализация сохраняет маленькую поверхность зависимостей и MIT-лицензию.

## D002 — модульный монолит

**Решение:** один CounterStrikeSharp plugin assembly, функции разделены внутренними модулями.

**Почему:** сейчас единый lifecycle полезнее независимой установки. Границы в коде позволяют вынести самостоятельную систему позже без раннего усложнения deployment.

## D003 — JSON по SteamID64 как offline fallback

**Решение:** host-mounted `config/skins.json` остаётся резервным источником до первого успешного ответа control plane для игрока; основной store — PostgreSQL.

**Почему:** сервер сохраняет базовую работоспособность при временной недоступности admin, но веб-панель и многопользовательская авторизация требуют транзакционного persistent store. Пустой remote response тоже авторитетен, иначе удалённые назначения появлялись бы снова из fallback.

## D004 — применение на lifecycle events

**Решение:** reload на connect/spawn; применение на connect/spawn/pickup через следующий кадр.

**Почему:** пользователь не хотел команд. Следующий кадр даёт игре завершить обновление pawn/inventory. File watcher и принудительное live-refresh сознательно не входят в v1.

## D005 — собственный Docker image

**Решение:** базовый Ubuntu image, официальный SteamCMD и архивы официальных релизов платформы.

**Почему:** не зависим от непрозрачного community CS2 image. Версии Metamod и CounterStrikeSharp закреплены, а CS2 обновляется SteamCMD при старте.

## D006 — LAN по умолчанию

**Решение:** пустой GSLT включает `sv_lan 1`; заданный токен включает публичный режим.

**Почему:** первый запуск должен работать локально без Steam account setup и случайного публичного exposure.

## D007 — отключение guideline guard

**Решение:** entrypoint создаёт CounterStrikeSharp `core.json` с `FollowCS2ServerGuidelines=false`.

**Почему:** CounterStrikeSharp иначе блокирует требуемые cosmetic operations. Это осознанный риск владельца сервера и должно оставаться заметным в README.

## D008 — gamedata для economy attributes

**Решение:** хранить Linux-сигнатуру `CAttributeList_SetOrAddAttributeValueByName` в собственном `gamedata/verona.json` и автоматически устанавливать файл вместе с плагином.

**Почему:** актуальные клиенты CS2 читают paint/seed/wear не только из fallback schema-полей, но и из economy attribute lists. Сигнатура нестабильна между обновлениями игры, поэтому она изолирована как явно поддерживаемый deployment artifact и требует smoke-test.

## D009 — веб-админка на Vue и PrimeVue

**Статус:** заменено D021.

**Решение:** Vue 3, TypeScript, Vite, PrimeVue/Aura и SignalR; frontend отдаётся ASP.NET Core backend.

**Почему:** веб-интерфейс работает на localhost и позднее удалённо без Electron runtime. PrimeVue предоставляет таблицы, диалоги и формы для администрирования, а один backend-контейнер упрощает deployment.

## D010 — PostgreSQL как единственный persistent store control plane

**Решение:** использовать существующий `postgres:18`; плагин получает данные только через HTTP API.

**Почему:** PostgreSQL подходит для банов, истории и будущих нескольких серверов. Изоляция DB-драйвера в backend сохраняет игровой плагин маленьким и не связывает его lifecycle с подключением к БД.

## D011 — CS2 запускается из панели

**Решение:** обычный Compose stack поднимает admin и PostgreSQL, а `cs2` находится в profile `game`. Панель запускает заранее созданный `verona-cs2-server` через Docker Engine API.

**Почему:** тяжёлый сервер не должен автоматически обновляться и занимать ресурсы при каждом старте Docker Desktop. Пользователь сначала выбирает параметры матча и только затем запускает игру.

## D012 — launch.env вместо изменения container environment

**Решение:** backend валидирует launch form и записывает `config/launch.env`, который entrypoint загружает перед формированием cfg и аргументов CS2.

**Почему:** Docker `start` не меняет environment существующего контейнера. Пересоздавать 60+ GB server container для каждой карты не нужно; маленький host-mounted snapshot даёт предсказуемый повторный старт. Поскольку файл исполняется Bash, добавлять в него можно только строго валидированные значения.

## D013 — Steam OpenID и server-side RBAC

**Решение:** Steam OpenID 2.0 — единственный вход. Роль хранится в PostgreSQL и проверяется на каждом запросе; `ADMIN_STEAM_IDS` используется только для bootstrap пустой БД. Backend разрешает обычным пользователям только `/api/me/**`.

**Почему:** OpenID подтверждает владение SteamID без хранения Steam credentials и без Web API key. DB-role меняется из панели с немедленным эффектом. Проверка прав только в browser UI была бы обходима прямым запросом, поэтому целевой SteamID self-service операций берётся из server-side session.

## D014 — статический каталог скинов, динамические назначения

**Решение:** публичные метаданные оружия и paint kits поставляются как `skins-catalog.json`, а выбор игрока хранится в PostgreSQL.

**Почему:** большой каталог редко меняется и эффективно отдаётся как static asset, тогда как назначения требуют авторизации и записи. Steam CDN хранит изображения; backend всё равно валидирует запрос, не доверяя каталогу в браузере.

## D015 — in-memory sessions на локальном этапе

**Решение:** opaque session tokens хранятся в памяти admin-процесса 12 часов и передаются HttpOnly cookie.

**Почему:** панель пока loopback-only и имеет один backend instance. Это уменьшает первоначальную сложность, но рестарт разлогинивает пользователей; перед масштабированием или публичным размещением сессии нужно перенести в shared persistent store и включить HTTPS/Secure cookies.

## D016 — собственная UI-система и loadout collections

**Решение:** компактный top navbar, нейтральная тёмная палитра с зелёным акцентом и игровые rarity colors. Skinchanger использует знакомую по CS2 арсенальную доску: глобальный переключатель T/CT, боковые слоты экипировки, колонки классов оружия и отдельную горизонтальную полосу ножей. Пользовательские коллекции хранят snapshots оружейных назначений в PostgreSQL.

**Почему:** navbar проще адаптируется к мобильному layout и будущим разделам. Rarity color несёт игровую семантику. Коллекции переключают loadout атомарно без копирования интерфейса или кода стороннего сайта.

## D017 — нож как единственный subclass loadout

**Решение:** хранить выбранную модель ножа canonical classname (`weapon_knife_*`/`weapon_bayonet`) в общей таблице скинов. Backend гарантирует одну модель на SteamID и team scope, а игровой модуль переводит штатный нож через `ChangeSubclass` только по allow-list item definition index.

**Почему:** paint kit сам по себе не меняет модель ножа. Отдельная таблица пока не нужна: API, коллекции и синхронизация уже работают с weapon key. Allow-list не позволяет превратить значение из БД в произвольный engine input. Реализация написана самостоятельно; GPL-код WeaponPaints не копируется.

## D018 — миграция на Metamod 2.0 (dev branch)

**Решение:** использовать Metamod:Source 2.0 development snapshots (из `mms.alliedmods.net/mmsdrop/2.0/`) вместо старых стабильных релизов 1.12.x с GitHub.

**Почему:** CS2 работает на Source 2 (engine 26) и требует Metamod 2.0. Версии 1.12.x были предназначены для Source 1, не имели структуры директорий `linuxsteamrt64` и приводили к полной неработоспособности Metamod. Свежие сборки ветки 2.0 также решают проблемы несовместимости ABI и отсутствующих символов (например, `UtlMemory_CalcNewAllocationCount`), возникающие после регулярных обновлений игры.

## D019 — перчатки и агенты как отдельные CT/T loadout slots

**Решение:** хранить перчатки и агентов отдельно от `player_weapon_skins`, по одному значению на команду. Перчатки содержат allow-listed диапазон item definition, paint/wear/seed; агент содержит строго валидированный `agents/models/*.vmdl`. Активные коллекции включают оба типа.

**Почему:** эти косметические объекты не являются оружием: перчатки живут в `CCSPlayerPawn.EconGloves`, а агент является моделью pawn. Раздельные таблицы не размывают инвариант canonical `weapon_*`, позволяют независимо валидировать engine input и сохраняют единый refresh/loadout поток skinchanger.

## D020 — командный scope оружейных скинов

**Решение:** ключ оружейного назначения включает точный `team` (`ct` или `t`). Общедоступное оружие имеет независимые CT/T варианты; пользовательское действие `both` атомарно записывает оба слота и не хранится как fallback. Командное оружие принимает только свой scope. Plugin всегда ищет точное назначение текущей команды.

**Почему:** это соответствует реальному CS2 loadout и не предлагает невозможные сочетания вроде AK-47 для CT. Материализация `both` устраняет скрытый приоритет fallback и гарантирует, что изменение одной стороны не удалит другую. Серверная проверка использует каталог доступности независимо от UI; старые записи `both` мигрируют в один или два точных слота в зависимости от доступности оружия.

## D021 — React, Tailwind CSS и shadcn/ui

**Решение:** frontend использует React 19, TypeScript, Vite и Tailwind CSS v4. Базовые controls генерируются официальным shadcn/ui CLI на Radix UI; Verona добавляет тему и композицию, но не дублирует существующие компоненты. Обычные HTML-кнопки остаются только у доменных интерактивных карточек карт и косметики.

**Почему:** локально генерируемые shadcn/ui-компоненты сохраняют доступность Radix UI и позволяют поддерживать собственный плотный игровой визуальный язык. Backend API, Steam OpenID, SignalR и server-side authorization boundaries при миграции не меняются.
